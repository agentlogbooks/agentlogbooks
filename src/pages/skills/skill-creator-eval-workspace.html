skill-creator eval workspace

Multi-table workspace for evaluating and iterating on skills.

git clone github.com/anthropics/skill-creator GitHub →

What it does

Each skill iteration produces eval results that you want to compare across versions — but a flat transcript can't answer "did iteration 8 actually get better than iteration 7?" The skill-creator eval workspace is the surface where versioned eval definitions, per-iteration results, and human feedback all join on stable IDs (eval_id, run_id, iteration_id). With those joins in place, regression queries, pass-rate trends, and reviewer notes become rows you can group, filter, and diff — instead of transcripts you have to re-read.

How it does it

01

Define an eval

Capture an eval as a versioned row: prompt, expected shape, success criteria. Each eval has a stable eval_id so it can be referenced across iterations.

SQLINSERT INTO evals (eval_id, name, prompt, expected_shape)
02

Iterate on the skill

Each skill version under test gets an iteration row, capturing what changed. The iteration_id is the join key that ties runs and feedback back to a specific revision.

SQLINSERT INTO iterations (iteration_id, skill_version, started_at, notes)
03

Run evals against the iteration

For each (eval × iteration), capture the run result with output, pass/fail, latency, and tokens. One row per pair — that's the grain that makes cross-iteration comparison work.

SQLINSERT INTO runs (run_id, eval_id, iteration_id, output, pass, latency_ms, tokens)
04

Aggregate per iteration

Group runs by iteration to compute pass rate, p50 latency, mean tokens. The aggregates are derived — they live in queries, not in a separate table you have to keep in sync.

SQLSELECT iteration_id, COUNT(pass)/COUNT(*) AS pass_rate FROM runs GROUP BY iteration_id
05

Capture human feedback

Reviewers attach notes to specific runs or iterations with severity. Feedback is its own table so it can be added asynchronously without rewriting run rows.

SQLINSERT INTO feedback (feedback_id, run_id OR iteration_id, reviewer, severity, note)
06

Pick winning iteration + ship

Query the data to pick the best iteration; ship the corresponding skill version. The decision is a query, not a vibe.

SQLSELECT * FROM iterations WHERE iteration_id = ( SELECT iteration_id FROM runs GROUP BY iteration_id ORDER BY pass_rate DESC LIMIT 1 )

Schema overview

TablePurpose
evals Eval definitions; versioned; stable eval_id
runs One row per (eval × iteration); run_id, pass, latency_ms, tokens, output
iterations One row per skill version under test; iteration_id, skill_version, notes
feedback Human notes on runs or iterations; severity, note, reviewer

Joins are the value. Without the workspace, "did this iteration get better?" is a transcript-reading task.
